# Sidebar Redesign: shadcn-vue Sidebar Component

**Date:** 2026-05-13
**Status:** Approved

## Goal

Replace the current custom `AppSidebar.vue` with the shadcn-vue Sidebar component system to standardize the UI library and restore the hierarchical nav structure from the old SPA (collapsible groups: HRM, Workflow, Recruitment, Request).

## Approach

Full replacement using shadcn-vue Sidebar primitives (Approach 1). Install via CLI, rewrite `AppSidebar.vue`, delete `NavItem.vue`, and wrap `layouts/admin.vue` with `SidebarProvider`.

## Data Model

```ts
type NavChild = {
  labelKey: string
  to: string
  module?: string
}

type NavItem = {
  labelKey: string
  icon: Component
  module?: string | null
  to?: string          // present for direct links
  children?: NavChild[] // present for collapsible groups
}
```

Permission filtering: a group is shown if the user is admin OR any of its children's modules pass `permissionStore.hasModule()`. A group with `children` is hidden entirely if no child is permitted.

## Sidebar Behavior

- `collapsible="icon"` on `<Sidebar>` ‚Äî collapses to icon-only rail
- When collapsed, hovering a collapsible group icon shows a floating popover with sub-items (shadcn-vue built-in behavior via `SidebarMenuSub`)
- `SidebarRail` renders a thin clickable strip on the edge to toggle collapsed state
- `SidebarTrigger` placed in `AppTopbar` for mobile/explicit toggle
- Active state: compare `route.path.startsWith(item.to)` ‚Äî apply `isActive` variant on `SidebarMenuButton` / `SidebarMenuSubButton`
- Collapsible open state: default open if any child route is active (`route.path.startsWith(child.to)`)

## Out of Scope

- Migrating `layoutStore.sidebarOpen` ‚Äî `useSidebar()` from shadcn-vue manages open state internally; `layoutStore` sidebar state is no longer needed after this change
- Dark mode adjustments beyond what shadcn-vue CSS variables already provide
- Any new nav items not present in old or current SPA
